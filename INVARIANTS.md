| ID | Rule | Tolerance | Status |
|---|---|---|---|
| **INV1** | Same-person leftover amounts match cook-day amounts exactly | 0 | hard |
| **INV2** | `calcTotals(p,d)` returns day kcal — must match sum of kcal across all slots from `getBalancedSlotIngredients(p,d,s)`. If it drifts, the daily stats bar shows different numbers than the cards add up to. | kcal within 2 | hard |
| **INV3** | `buildShoppingList(trip, who)` sums ingredient amounts per trip — must match the balanced cache summed across those days/people. If it drifts, the shopping list says "buy 2 lbs chicken" but the cards only eat 1.5 lbs. | exact | hard |
| **INV4** | Any solo pkg meal (not shared, not in leftover batch) must have all its pkg ingredients in `PKG_FLEX_CONFIG` so `applyTripFlexScaling` can resolve the trip-level package waste. Replaced the old Phase 1.5 inline cleanup — enforced as a hard invariant now so adding a new pkg ingredient without a flex entry fails fast. | exact | hard |
| **INV5** | Card macros (rendered on each slot's UI via `computeCardMacros`) must match `getBalancedSlotIngredients(p,d,s)` for that slot. If it drifts, a card shows "450 kcal" but the ingredient list implies 480. | exact | hard |
| **INV6** | Per-meal protein/carb ratio changes by less than 50% vs base recipe (signal that day-balancer swaps aren't distorting the original dish too aggressively) | ~2/run typical | **tracking-only** |
| **INV7** | Cross-person cook scoop ratio consistent across all ingredients. For a shared/leftover batch, every ingredient's per-person split must be proportional to each person's kcal share of the batch — one pot, one ratio. | 1% | hard |
| **INV8** | Solo per-serving amounts AND batch totals land on ≤2 decimals OR 1/8 fractions (0.125, 0.375, 0.625, 0.875). Prevents weird 3-decimal display values like 7.701. pkg items exempt. | 0.001 | hard |
| **INV9** | Him/Her tabs never show "Combined ingredients (serves N)" header for a same-day shared meal. The Shared tab owns the combined view; Him/Her tabs must render per-serving only. | exact | hard |
| **INV10** | Every lunch/dinner recipe has greater than 0 cups of veg (static check over the MEALS array, catches recipes that accidentally have no vegetable component) | static check | hard |
| **INV11** | At least 2-day gap between batches of the same meal within a week (excludes `noLeftover` recipes). Prevents the randomizer from cooking the same dish three days in a row. | exact | hard |
| **INV12** | `lo.totalServings === lo.portions.length` AND every portion in a batch shares the anchor's meal ID. Catches detector drift — if the leftover/batch grouping ever produces inconsistent state, this fires. | exact | hard |
| **INV13** | Per-serving amount for any ingredient with `db.minAmt` or `db.maxAmt` defined must fall within those bounds, for solo AND batch portions. Solo slots (no leftover-map entry) additionally check `db.minAmtSolo` when defined, enforcing a higher floor for single-portion cooks (pan-oil realism, aromatic quality). | 0.001 | hard |
| **INV14** | **Household-level**: no two NEW cooks of the same meal (by EITHER person) should be within 5 days. Leftovers of an existing cook don't count (same batch anchor). Shared cooks count once via the anchor. Solo slots ARE cooks. **Breakfast and snack exempt** — small pools (9-of-27 breakfast, 17-of-21 snack) make a 5-day gap mathematically impossible. Will revisit when breakfast normalization grows the pool. Promoted from per-person to household 2026-04-21 along with matching variety-filter change. | exact | **tracking-only** |
| **INV15** | Count of **time-shifted** lunch/dinner leftovers **him** eats (regardless of who cooked the batch). Excludes `sameDayShared` — those are the other-person portion of a same-day shared cook, not a reheated leftover. Emits one line per run with `count=N`. Aggregated by MPStress as `avgLeftoversEaten.him`. | — | **tracking-only** |
| **INV16** | Same as INV15 but for **her**. `sameDayShared` excluded (fixed 2026-04-21; previously inflated Her's count by 4-5/wk). Aggregated by MPStress as `avgLeftoversEaten.her`. | — | **tracking-only** |
| **INV17** | Balancer↔calcTotals kcal canary: reconstruct what `balanceDayMacros.dailyMacros()` would see (cache slots × same-day-cook multiplicity + eat-outs + late snack, **gating SKIPPED/EAT_OUT slots like calcTotals** — V234) and compare against `calcTotals(p,d)` per person-day. Catches `sameDayCookServings` double-count/under-count bugs in post-pipeline re-runs. | within 2 kcal | hard |
| **INV18** | `runBalanceAdjusters` convergence-loop cap-hit rate: per randomize, ≤10% of calls may hit the 6-iteration safety cap. A higher rate signals a new oscillation source between the idempotent adjusters. Debug via `window._rbaDiagEnabled = true` → `window._rbaDiag`. | ≤10% per run | **tracking-only** |
| **INV19** | Cooked/dry DB consistency: every "cooked"-named entry has `pkg` (canned) or `dry` (cup-cooked→cup-dry conversion); every cup-unit "dry" entry has a cooked sibling. Spices/herbs excluded by the cup-unit filter. Static check over NUTRI_DB. | static | hard |
| **INV20** | Pkg/produce **trip waste** threshold: pkg waste >1% of bought, produce waste >50% of one `perWhole`. **Hard count = INV20 − INV20Soft.** Two evaluation sites (per-trip + weekly `_inv20CarryWeekly` for `crossTripCarry` items), both gating `longShelfLife`/`acceptableWaste`/`nonWaste` (weekly `longShelfLife` gate added R14-L). | >1% pkg / >50% produce | hard |
| **INV20Soft** | Subset of INV20 fires for `softWaste:true` items (marinara, broth; banana/avocado/orange/jalapeño produce). Subtracted from the hard count — user-accepted waste, visible but not a failure. | — | **tracking-only** |
| **INV20Leftover** | Forward-inventory line: this week's cook2 partial of a `crossTripCarry` item that becomes next week's `priorCarry`. Informational, never a fire (excluded from every hardFail filter — V207 formatReport, R14 saveBaseline). | — | **tracking-only** |
| **INV21** | Pkg-only split of INV20 (waste >1% of bought). | >1% | **tracking-only** |
| **INV22** | Produce-only split of INV20 (waste >50% of one whole). Lax threshold — produce sizes are structurally coarse. | >50% | **tracking-only** |
| **INV23** | Per-row displayed macros sum to the card header per-serving × N for combined views. Gate mirrors `cardShowsCombinedHeader` (same-person `servings>1` OR `crossPersonCook` — widened V235). Cross-person branch verifies the RENDERED per-portion-summed rows (idx-matched) against unrounded cache truth. | solo 0; batch `ceil(N/2)`; xp `ceil(rows/2)` | hard |
| **INV24** | State-mutation mutual exclusion: `SKIPPED∧EAT_OUT` same slot; `SHARED='shared'∧SKIPPED/EAT_OUT` either side; `'eo-{side}'∧SKIPPED` same side; `'eo-both'∧SKIPPED` either side. Catches stale sync / pre-V31 data. | exact | hard |
| **INV25** | Card dropdown display coherence: an option within 0.02 of the stored amount exists AND its RENDERED label round-trips via `_parseQtyLabel` within the user-approved 0.0601 fmtFrac glyph band AND no other option renders an identical label for a different value (de-tautologized V235 — the old value-only check could never fire). | 0.02 val / 0.0701 label | hard |
| **INV26** | Stress-harness guard: hard-fails any stress test starting on polluted (non-pristine) state. Not a runtime-app invariant — protects against false baselines. | exact | hard (harness) |
| **INV27** | NUTRI_DB kcal↔macro consistency: kcal ≈ 4·pro + 4·carb + 9·fat per entry. Fires when \|diff\| > 25 kcal AND relative deviation > 25% — empirical threshold that separates real bugs (basis mismatch, swapped/typo'd fields) from legit label noise (fiber-heavy veg, citrus, trace-kcal condiments). Also fires on negative/missing kcal. Mirrored by a confirm() in the New Ingredient form. | 25 kcal ∧ 25% | hard (static) |
